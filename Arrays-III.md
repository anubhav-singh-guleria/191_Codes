<div class="elfjS" data-track-load="description_content"><p>You are given an <code>m x n</code> integer matrix <code>matrix</code> with the following two properties:</p>

## Arrays III Pow(x, n)
<div class="elfjS" data-track-load="description_content"><p>Implement <a href="http://www.cplusplus.com/reference/valarray/pow/" target="_blank">pow(x, n)</a>, which calculates <code>x</code> raised to the power <code>n</code> (i.e., <code>x<sup>n</sup></code>).</p>

<p>&nbsp;</p>
<p><strong class="example">Example 1:</strong></p>

<pre><strong>Input:</strong> x = 2.00000, n = 10
<strong>Output:</strong> 1024.00000
</pre>

<p><strong class="example">Example 2:</strong></p>

<pre><strong>Input:</strong> x = 2.10000, n = 3
<strong>Output:</strong> 9.26100
</pre>

<p><strong class="example">Example 3:</strong></p>

<pre><strong>Input:</strong> x = 2.00000, n = -2
<strong>Output:</strong> 0.25000
<strong>Explanation:</strong> 2<sup>-2</sup> = 1/2<sup>2</sup> = 1/4 = 0.25
</pre>

<p>&nbsp;</p>
<p><strong>Constraints:</strong></p>

<ul>
	<li><code>-100.0 &lt; x &lt; 100.0</code></li>
	<li><code>-2<sup>31</sup> &lt;= n &lt;= 2<sup>31</sup>-1</code></li>
	<li><code>n</code> is an integer.</li>
	<li>Either <code>x</code> is not zero or <code>n &gt; 0</code>.</li>
	<li><code>-10<sup>4</sup> &lt;= x<sup>n</sup> &lt;= 10<sup>4</sup></code></li>
</ul>
</div>

### Solution
<div class="FN9Jv WRmCx"><h4 id="intuition">Intuition</h4>
<p>This problem can be easily solved using <strong>Binary Exponentiation</strong>.</p>

<h4 id="approach">Approach</h4>
<p>Initialize ans as 1.0  and store a duplicate copy of n i.e nn using to avoid overflow</p>
<p>Check if nn is a negative number, in that case, make it a positive number.</p>
<p>Keep on iterating until nn is greater than zero, now if nn is an odd power then multiply x with ans ans reduce nn by 1. Else multiply x with itself and divide nn by two.</p>
<p>Now after the entire binary exponentiation is complete and nn becomes zero, check if n is a negative value we know the answer will be 1 by and.</p>

<h4 id="complexity">Complexity</h4>
<ul>
<li>Time complexity:O(log(n))</li>
</ul>

<ul>
<li>Space complexity:O(1)</li>
</ul>

## Arrays III Majority Element
<div class="elfjS" data-track-load="description_content"><p>Given an array <code>nums</code> of size <code>n</code>, return <em>the majority element</em>.</p>

<p>The majority element is the element that appears more than <code>⌊n / 2⌋</code> times. You may assume that the majority element always exists in the array.</p>

<p>&nbsp;</p>
<p><strong class="example">Example 1:</strong></p>
<pre><strong>Input:</strong> nums = [3,2,3]
<strong>Output:</strong> 3
</pre><p><strong class="example">Example 2:</strong></p>
<pre><strong>Input:</strong> nums = [2,2,1,1,1,2,2]
<strong>Output:</strong> 2
</pre>
<p>&nbsp;</p>
<p><strong>Constraints:</strong></p>

<ul>
	<li><code>n == nums.length</code></li>
	<li><code>1 &lt;= n &lt;= 5 * 10<sup>4</sup></code></li>
	<li><code>-10<sup>9</sup> &lt;= nums[i] &lt;= 10<sup>9</sup></code></li>

<p><strong>If</strong> we don't know weither the majority element exists, the algorithm is two pass: the first one chooses the best candidate that can potentially be a majority element, the second pass confirms or rejects this guess.<br>
On the first pass we focus on one element (leader) at a time and maintain the counter that equals the number of occurrences of the leader minus the number of occurences of all the other elements. When the counter drops to<code> 0</code> we focus on the next element. The last elemet we were focused on will be the candidate to the title of the majority element. We need to confirm in a separate array traversal.</p>
<p><strong>Explanation 1: runs of 50 %</strong></p>
<p>If we think about it our algorithm breaks our array into segments (I call them runs). In each run the first element (leader) of the run "dominates" all the other elements. "Dominates" means that it occurs more times than all the others elements in the run combined. We terminate the run when all the other elements combined occur as much times as the leader (<code>cnt == 0</code>).<br>
<br><br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584534212.png" alt="image"><br>
<br></p>
<p>This tells us that the leader as well as <strong>every other element</strong> occurs <strong>50% of maybe less</strong> in the run. The same is true if we consider few consecutive runs stacked together: in each of them every element can occur 50% or less that's why in all of them combined the same is true. The last run can be unfinished. In all previous runs combined no element occured more than 50% that's why <strong>if</strong> the array has the majority element this element necessarily <strong>has to be</strong> the leader of the last run and has to occupy <strong>more</strong> than 50% of it (otherwise the last run would not be the last or the array wouldn't have the majority element).<br>
<br><br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584543475.png" alt="image"><br>
<br><br>
So in fact all the decisive staff occurs in the last run, but we have to do all the other runs to determine when the last runs starts and what element will be its leader<br>
<br><br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584543732.png" alt="image"><br>
<br><br>
Note that there could be many small 2-element runs or one giant run<br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584538411.png" alt="image">so we can't say that we are reducing our problem into a smaller one. Breaking down into runs is kinda 'magical' as at the end it leaves us with the run whose leader is majority element of the initial array (if it existed).</p>
<p><strong>Explanation 2: pairing</strong></p>
<p>This point of view will be helpful for the generalisation of the problem <a href="https://leetcode.com/problems/majority-element-ii/" target="_blank">229. Majority Element II</a>.<br>
Suppose we have an array of numbers and we want to make pairs of <strong>different</strong> elements. This means that a pair say <code>(1,2)</code>is valid while <code>(1,1)</code> is not.<br>
If all the elements in the array were unique we would traverse the array and pair up each element with the next one.<br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584538862.png" alt="image"><br>
<br><br>
We run into problem if the array has repeating elements. Now we have to "save" (using a stack perhaps) the excess element<br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584539197.png" alt="image"></p>
<p>and fastforward to the element different from the saved one.<br>
<img src="https://assets.leetcode.com/users/andriy111/image_1584539429.png" alt="image"></p>
<p>We do this untill the stack containing excess element is empty, after that we can fill it with another excess element if we'll encounter it.<br>
In fact the stack contains the same element multiple times so there's actually no need in the stack: we can store the element and it's abundance in two variables <code>c</code> and <code>cnt</code>.<br>
Thus with our algorithm we can maintain production of pairs of different elements, the <code>c</code> variable stores the element that is currently in excess and <code>cnt</code> stores the excess itself. Pairing of elements is useful because the leftover element after we reach the end of the array will be the majority element (if it exists). Let me explain why. All in all we will have <code>⌊ n/2 ⌋</code> pairs and even if majority element was present in each of them we would still have some of if left (by definition majority should occur <strong>more</strong> than <code>⌊ n/2 ⌋</code> times in the array). If majority occures not in each pair, we will have even more unpaired instances of it.</p>

### Solution 
<div class="break-words"><div><div class="FN9Jv WRmCx"><p>For those who aren't familiar with Boyer-Moore Majority Vote algorithm,<br>
I found a great article (<a href="http://goo.gl/64Nams" target="_blank">http://goo.gl/64Nams</a>) that helps me to understand this fantastic algorithm!!<br>
Please check it out!</p>
<p>The essential concepts is you keep a counter for the majority number <strong>X</strong>. If you find a number <strong>Y</strong> that is not <strong>X</strong>, the current counter should deduce 1. The reason is that if there is 5 <strong>X</strong> and 4 <strong>Y</strong>, there would be one (5-4) more <strong>X</strong> than <strong>Y</strong>. This could be explained as "4 <strong>X</strong> being paired out by 4 <strong>Y</strong>".</p>
<p>And since the requirement is finding the majority for more than ceiling of [n/3], the answer would be less than or equal to two numbers.<br>
So we can modify the algorithm to maintain two counters for two majorities.</p>
<p>Code</p>

```
#include <vector>

class Solution {
public:
    std::vector<int> majorityElement(std::vector<int>& nums) {
        std::vector<int> result;
        if (nums.empty()) {
            return result;
        }

        int count1 = 0, count2 = 0;
        int candidate1 = 0, candidate2 = 1;

        for (int n : nums) {
            if (n == candidate1) {
                count1++;
            } else if (n == candidate2) {
                count2++;
            } else if (count1 == 0) {
                candidate1 = n;
                count1 = 1;
            } else if (count2 == 0) {
                candidate2 = n;
                count2 = 1;
            } else {
                count1--;
                count2--;
            }
        }

        count1 = 0;
        count2 = 0;

        for (int n : nums) {
            if (n == candidate1) {
                count1++;
            } else if (n == candidate2) {
                count2++;
            }
        }

        if (count1 > nums.size() / 3) {
            result.push_back(candidate1);
        }
        if (count2 > nums.size() / 3) {
            result.push_back(candidate2);
        }

        return result;
    }
};
```

<p>✔️ <em><strong>Solution - II (Dynamic Programming - Memoization)</strong></em></p>
<p>The above solution had a lot of redundant calculations. There are many cells which we reach multiple times and calculate the answer for it over and over again. However, the number of unique paths from a given cell <code>(i,j)</code> to the end cell is always fixed. So, we dont need to calculate and repeat the same process for a given cell multiple times. We can just store (or memoize) the result calculated for cell <code>(i, j)</code> and use that result in the future whenever required.</p>
<p>Thus, here we use a 2d array <code>dp</code>, where <code>dp[i][j]</code> denote the number of unique paths from cell <code>(i, j)</code> to the end cell <code>(m-1, n-1)</code>. Once we get an answer for cell <code>(i, j)</code>, we store the result in <code>dp[i][j]</code> and reuse it instead of recalculating it.</p>

<p>✔️ <em><strong>Solution - III (Dynamic Programming - Tabulation)</strong></em></p>
<p>We can also convert the above appraoch to an iterative version. Here, we will solve it in bottom-up manner by iteratively calculating the number of unique paths to reach cell <code>(i, j)</code> starting from <code>(0, 0)</code> where <code>0 &lt;= i &lt;= m-1</code> and <code>0 &lt;= j &lt;= n-1</code>. We will again use dynamic programming here using a <code>dp</code> matrix where <code>dp[i][j]</code> will denote the number of unique paths from cell <code>(0, 0)</code> the cell <code>(i, j)</code>. (Note this differs from memoization appraoch where <code>dp[i][j]</code> denoted number of unique paths from cell <code>(i, j)</code> to the cell <code>(m-1,n-1)</code>)</p>
<p>In this case, we first establish some base conditions first.</p>
<ul>
<li>We start at cell <code>(0, 0)</code>, so <code>dp[0][0] = 1</code>.</li>
<li>Since we can only move right or down, there is only one way to reach a cell  <code>(i, 0)</code> or <code>(0, j)</code>. Thus, we also initialize <code>dp[i][0] = 1</code> and <code>dp[0][j]=1</code>.</li>
<li>For every other cell <code>(i, j)</code> (where <code>1 &lt;= i  &lt;= m-1</code> and <code>1 &lt;= j &lt;= n-1</code>), we can reach here either from the top cell <code>(i-1, j)</code> or the left cell <code>(i, j-1)</code>. So the result for number of unique paths to arrive at <code>(i, j)</code> is the summation of both, i.e, <code>dp[i][j] = dp[i-1][j] + dp[i][j-1]</code>.</li>
</ul>

<p>✔️ <em><strong>Solution - IV (Space Optimized Dynamic Programming)</strong></em></p>
<p>In the above solution, we can observe that to compute the <code>dp</code> matrix, we are only ever using the cells from previous row and the current row. So, we don't really need to maintain the entire <code>m x n</code> matrix of <code>dp</code>. We can optimize the space usage by only keeping the current and previous rows.</p>
<p>A common way in <code>dp</code> problems to optimize space from 2d dp is just to convert the <code>dp</code> matrix from <code>m x n</code> grid to <code>2 x n</code> grid denoting the values for current and previous row. We can just overwrite the previous row and use the current row as the previous row for next iteration. <ul>
<li>We start at <code>(0, 0)</code> cell and move to <code>(m-1, n-1)</code> cell.</li>
<li>We need to make <strong><code>m-1</code> down-moves</strong> and <strong><code>n-1</code> right-moves</strong> to reach the destination cell.</li>
<li>Thus, we need to perform a <strong>total number of <code>m+n-2</code> moves</strong>.</li>
<li>At each cell along the path, we can choose either the right-move or down-move and we need to find the number of unique combinations of these choices (which eventually leads to unique paths).</li>
<li>This is nothing but calculating the <strong>number of different ways to choose <code>m-1</code> down-moves and <code>n-1</code> right-moves from a total of <code>m+n-2</code> moves</strong>. Mathematically, this can be represented as -</li>

## Arrays III Reverse Pairs
<div class="elfjS" data-track-load="description_content"><p>Given an integer array <code>nums</code>, return <em>the number of <strong>reverse pairs</strong> in the array</em>.</p>

<p>A <strong>reverse pair</strong> is a pair <code>(i, j)</code> where:</p>

<ul>
	<li><code>0 &lt;= i &lt; j &lt; nums.length</code> and</li>
	<li><code>nums[i] &gt; 2 * nums[j]</code>.</li>
</ul>

<p>&nbsp;</p>
<p><strong class="example">Example 1:</strong></p>

<pre><strong>Input:</strong> nums = [1,3,2,3,1]
<strong>Output:</strong> 2
<strong>Explanation:</strong> The reverse pairs are:
(1, 4) --&gt; nums[1] = 3, nums[4] = 1, 3 &gt; 2 * 1
(3, 4) --&gt; nums[3] = 3, nums[4] = 1, 3 &gt; 2 * 1
</pre>

<p><strong class="example">Example 2:</strong></p>

<pre><strong>Input:</strong> nums = [2,4,3,5,1]
<strong>Output:</strong> 3
<strong>Explanation:</strong> The reverse pairs are:
(1, 4) --&gt; nums[1] = 4, nums[4] = 1, 4 &gt; 2 * 1
(2, 4) --&gt; nums[2] = 3, nums[4] = 1, 3 &gt; 2 * 1
(3, 4) --&gt; nums[3] = 5, nums[4] = 1, 5 &gt; 2 * 1
</pre>

<p>&nbsp;</p>
<p><strong>Constraints:</strong></p>

<ul>
	<li><code>1 &lt;= nums.length &lt;= 5 * 10<sup>4</sup></code></li>
	<li><code>-2<sup>31</sup> &lt;= nums[i] &lt;= 2<sup>31</sup> - 1</code></li>
</ul>
</div>
